    G6, ONE troubleshooting⌁

    See how to set up G6/ONE first if you haven't already. This page covers common problems once it's connected, or preventing it from connecting at all.

    No connectivity at all⌁

    1. Start from the recommended settings and disable engineering mode.
    2. Communication with the transmitter happens once every 5 minutes — after changing any setting, wait for a full read cycle before judging its effect on the status page.
    3. Make sure the transmitter is activated, and that no other phone or device is currently set to collect from it (disconnect it if there is).
    4. Check the Dex Status page. If there's a command stuck in the queue, it should clear within the next read cycle — if it doesn't, clear the queue.
    5. If the status page shows "Deep sleeping errors" or "Checking Auth errors" with a "Last Connected" time that's consistently under 5 minutes, tap Forget Device on the Classic Status page, go back to the Dex Status page, and approve the new pair request that follows (on Android 9 or older you won't get a visible prompt — it pairs automatically). If it still won't connect, trigger a pair request manually.
    6. If there's no disconnect and the status page shows "OK ** days" for sensor status but you're getting no readings, make sure the xDrip session itself is started — tap Start Sensor and enter the calibration code shown on the status page (this doesn't affect an already-running G6 session).
    7. If it's still not working, disable collection in xDrip, install the Dexcom app, and try it there instead. If the Dexcom app connects fine, please report it — that tells us it's an xDrip-side issue, not the transmitter.

    If you have connectivity and readings but it drops out occasionally, see Intermittent connectivity instead.

    Intermittent connectivity⌁

    xDrip scans every 5 minutes to stay connected, running in the background the rest of the time to save battery. Newer Android versions (11+) and phone-manufacturer power-saving features can interfere with this, which is why initial connectivity trouble is common on recent Android.

    • Make sure you're on at least the current stable xDrip release, and start from the recommended settings.
    • Check for phone-brand-specific guidance at dontkillmyapp.com.
    • Pixel — disabling battery optimization needs more than "Allow background usage": tap the note shown and choose Unrestricted.
    • Xiaomi — battery optimization may silently re-enable itself after every reboot; you may need to disable it again each time.
    • Unihertz — Android Settings → Smart Assist → Power Save Manager, and whitelist xDrip.
    • Cubot (e.g. KingKong Mini) — open the bundled DuraSpeed app and remove xDrip from its power-saving restrictions.

    If disconnects continue, try these one at a time: enable Special Pairing Workaround (can help even on non-Samsung phones), try disabling Minimize Scanning, and on Samsung devices try enabling Trust Auto-Connect in Bluetooth Settings.

    Understanding the Dex Status page⌁

    Open it from the top-left menu → System Status, then swipe to the "Dex Status" page.

    Connectivity is OK when: there's a "Last Connected:" line, its value stays under 5 minutes while you're near the phone, it climbs and rolls back to 0 every ~5-minute cycle, and there are no messages like "Hunting Transmitter", "Scanning Error", "Waiting Connect Errors", "Checking Auth Errors" or "Deep Sleeping Errors". Any of those messages, or a "Last Connected" value that doesn't behave as above, means a disconnect — resolve it before issuing any command (start, stop, calibrate, hard reset).

    xDrip reflects what the transmitter reports once every 5 minutes (a sample-and-hold system), so after any change or command, give it a full cycle (or two) before drawing conclusions — don't judge by checking the status page every few seconds.

    Picking up a session already started elsewhere

    If a session is already running (started from another phone, app, or a medical device) you don't need to start a new one in xDrip — just get connectivity working as above and be patient, it can take 10–15 minutes to pick up. This doesn't apply to G7/ONE+ (those always auto-start). For G6/ONE, if the calibration code shown on the Dex Status page looks wrong or missing, ignore it — it's the transmitter that needs the correct code, not xDrip, and only matters again if you're starting a genuinely new sensor.

    The transmitter ID shown on this page is unique to your transmitter — be careful who you share it with, and black it out of screenshots before posting them for help; no one needs to see it to help you.

    Sensor won't start / "sensor failed" messages⌁

    Most of these come down to the transmitter not being confident this is a genuine new sensor start:

    • The transmitter must be out of any sensor for at least 10 minutes before you start a new one. A common mistake is applying the new sensor while the old one is still running, stopping the old one, and immediately snapping the transmitter into the new sensor — since it was never out for 10 minutes, the transmitter treats it as a restart attempt and fails the new sensor with an error.
    • Make sure you have proper connectivity before attempting to start — if you don't, resolve that first rather than retrying start/stop.
    • If you get a "sensor start failed" message, wait it out rather than repeatedly retrying — repeated attempts without addressing the underlying connectivity or timing issue tend to make it worse.

    Transmitter queue stuck⌁

    The transmitter is only listening for a brief moment once every 5 minutes; the rest of the time it's silent to save battery. Any command you issue (Start Sensor, Stop Sensor, Calibrate) sits in a queue until the next window — you'll see it listed on the Dex Status page. As long as it's listed, the transmitter hasn't received it yet, so don't issue another command on top of it, and especially don't issue commands while you don't have proper connectivity — you can end up locked out entirely.

    If a command has been sitting in the queue for longer than 5 minutes and you can't establish connectivity, clear it:

    1. Tap the dropper (treatment) symbol on the main screen.
    2. Long-press the microphone icon at the bottom right to get a text entry field instead of voice input.
    3. Type clear transmitter queue and confirm.

    Don't clear the queue just because a command is sitting there normally after you issued it — that's expected and clearing it would stop the command from ever being applied.

    Triggering a pair request manually⌁

    1. Settings → Dexcom Transmitter ID. Note the current value (write it down or screenshot it).
    2. Change the last character to anything else and apply.
    3. Change it back to the correct value and apply.
    4. Go to the Dex Status page — you should see "Hunting Transmitter" in red.
    5. Within 5 minutes you should get a pair request on Android 10+ — approve it. On Android 9 or older, it pairs automatically with no visible prompt.
    6. If your screen is locked when the request notification arrives, unlock it, pull down the notification shade, and approve it from there.

    How strong is the transmitter signal?⌁

    If the transmitter isn't actually broadcasting, no xDrip configuration will fix that — a Bluetooth scanner lets you check independently of xDrip.

    1. Install a continuously-scanning Bluetooth scanner app (e.g. "BLE Scanner") on any Android phone — it doesn't need to be the one running xDrip. Watch out for on-screen ads in these apps.
    2. Grant the permissions it asks for, keep the transmitter in the same room, and let it scan.
    3. Look for a device named Dexcom** (last two characters of the transmitter's serial number) for G6/ONE, or DXCM**/DX02** (random suffix) for G7/ONE+.
    4. Note the negative signal-strength number next to it — you may need to wait up to 5 minutes for a bold (current) listing to appear.
    5. If it never appears, the transmitter isn't transmitting at all. If it does, report the device name and the number to whoever is helping you debug.

    How good is the battery?⌁

    Transmitter batteries degrade in storage even when unused — that's why the box shows both a manufacture date and an expiry date. On the Dex Status page you'll see two voltages, Voltage A and Voltage B. They vary between individual transmitters, but as a battery nears end of life, Voltage B starts a steady, continuous decline — worth keeping an eye on over a transmitter's life if readings start looking unreliable near the expected end of its battery life.

    Transmitter warranty⌁

    A transmitter is under warranty if both are true: the Use-By/Ship-By date on the box hadn't passed when you started (or received) it, and you started using it within 5 months of receiving it. Coverage is for 90 uninterrupted days of proper operation. Because of this, it's not worth stockpiling transmitters — the battery ages regardless, and the warranty clock can run out before you get to use a spare.

    Hard reset⌁

    G6 can be hard reset through xDrip only if its firmware is 1.6.5.25 or earlier — newer firmware can't be hard reset this way. A hard reset zeroes the transmitter's day counter on the status page; do it only while there's no active session (e.g. after stopping a sensor, before starting the next one).

    1. Confirm you have proper connectivity to the transmitter, and disable the Dexcom app (xDrip and Dexcom app can't both collect from the same transmitter).
    2. On the main screen, tap the dropper (treatment) symbol.
    3. Long-press the microphone icon at the bottom right to get a text field.
    4. Type Hard Reset Transmitter and confirm — you'll see a message that the attempt will be made. Don't close xDrip; the command still needs to reach the transmitter.
    5. Watch the status page — it can take up to 5 minutes to submit. You may see a warning that it may have failed; if so, restart your phone or force-stop and restart xDrip.
    6. Remove the transmitter from Android's list of previously connected Bluetooth devices.

    If you intend to keep using the transmitter afterward, activate it again as if it were brand new.

    Restarting a sensor beyond its normal session⌁

    Read this whole section before attempting it

    Restarting is not approved by the manufacturer and carries real risk. Readings will be significantly (and predictably) too high right after the restart's warm-up — if you correct that with insulin, or run a closed loop without opening it first, you can cause hypoglycemia.

    If the transmitter's firmware (shown on the Dex Status page) is 1.6.5.25 or earlier, you can use xDrip's preemptive restart instead and skip all of this — it restarts the sensor for you automatically. Otherwise, if you want to keep the same sensor running past its normal 10-day session, this is how, using the original calibration code (attempting a no-code restart has more reports of malfunction):

    1. Make sure connectivity is solid first — restarting won't fix an underlying connectivity problem, and you don't have to restart before day 10 anyway, only if you choose to.
    2. Tap Stop Sensor (top-left menu), even if it's already stopped and Stop Sensor is the only option shown. Wait for it to clear the queue on the status page (~5 minutes) before continuing — you'll then see "Sensor: Stopped".
    3. Remove the transmitter without damaging the sensor. Don't wipe off the petroleum jelly on the contacts.
    4. Wait at least 10 minutes with the transmitter out.
    5. Snap the transmitter back in and Start Sensor, entering the original calibration code again.

    After the 2-hour warm-up, expect readings to run significantly high — this is because the transmitter now thinks it's on day 1 of a fresh sensor, when your body's actual response to the sensor is that of a several-days-old one. You'll need to calibrate carefully: correct by no more than 20% at a time, spacing calibrations at least 25 minutes apart (45 for a third), and only while the trend is flat — otherwise you risk a confused calibration error. You'll see "Received backfill packet out of sequence" in the logs during this; that's expected and can be ignored. As with any aging sensor, keep checking against a glucose meter occasionally afterward.

    Changing the Bluetooth slot (advanced, not recommended)⌁

    A G6 transmitter has two communication slots — that's how you can use a t:slim pump and see readings in xDrip, or use a Dexcom receiver and xDrip together. xDrip defaults to slot 2; a t:slim pump and the Dexcom receiver both use slot 1.

    Warning

    This setting is still behind engineering mode because testing has produced unexpected results, including transmitters that were permanently disabled. It's not currently recommended.

    If you still want to change it: enable engineering mode, then Settings → G5/G6 Debug Settings → "Manual Slot Number", and set it to 1 to force the non-default slot (disabling engineering mode reverts to the default). Two collector apps using the two different slots can't run on the same phone.

    Removing a transmitter without damaging the sensor⌁

    Normally, once you peel off a sensor you open it up, take the transmitter out, and throw the sensor away. This is how to remove the transmitter while leaving the sensor intact and still on your body, e.g. to move it to a fresh sensor.

    Don't touch the contacts on either the transmitter or the sensor while doing this, and avoid tight clothing over the sensor in the meantime, so you don't accidentally wipe away the petroleum jelly protecting the contacts.

    The sensor has two prongs that latch onto the sides of the transmitter, held under spring pressure by the flexible material inside the sensor housing. To release them, you need something thin and sturdy to gently push each prong away from the transmitter — cut a small strip from the plastic packaging of a spare sensor and round its corners with scissors to make a simple tool (keep it away from children or pets — it's a choking hazard in small hands).

    Gently insert the wider, rounded end of the strip between the sensor and the transmitter, right where a prong is, until you feel or hear it click and release. The compressed material inside then pushes the transmitter away on that side by itself — you don't need to pry further, just avoid pressing down on top of the transmitter, which would re-seat it. Repeat on the other side, and the transmitter lifts free.

    Petroleum jelly around the contacts⌁

    After inserting a sensor and before snapping in the transmitter, you may see small blobs of petroleum jelly around the contacts inside the sensor — leave them alone, and keep dust and anything else out of the sensor. If there's a little bleeding, dab it with a Q-tip until it stops rather than letting blood reach the contacts, since blood can compromise their insulation and cause erroneous low readings.

    Cleaning and snapping in the transmitter spreads the jelly around the contacts, forming a moisture barrier. When removing a transmitter for reuse, don't insert anything like a test strip between transmitter and sensor to do it — that can wipe the jelly away — and once the transmitter is out, avoid touching its contacts or wearing tight clothing over the site until you snap it back in.

    What not to do⌁

    • Don't remove a transmitter from its box, or touch its contacts, before you're ready to use it — the day counter starts immediately and can't be paused.
    • Don't throw away a sensor's adhesive cover before noting the calibration code printed on it.
    • Don't pre-soak a sensor (inserting it well before starting it in xDrip) — it interferes with factory calibration.
    • Don't start a sensor unless the transmitter has been out of any sensor for at least 10 minutes first.
    • Don't correct more than 20% of a reading in a single calibration.
    • Don't stop a sensor just because you think you can easily restart it — there's almost always a better option than stopping.
    • When you switch phones, remember to disable collection on the old one, or both phones will fight over the connection.
    • Don't attempt a restart without understanding and accepting the risks — it isn't manufacturer-approved and readings will be wrong for a while afterward.

    OB1 debugging logs⌁

    If a developer asks you for detailed logs while troubleshooting a G6/G7/ONE/ONE+ connection issue:

    • Enable: scan the OB1 debugging QR code via Settings → Auto configure, or manually add Ob1G5CollectionService:v,Ob1G5StateMachine:v under Settings → Less common settings → Extra Logging Settings → Extra tags for logging.
    • Wait for the problem to actually happen before uploading — enabling it alone won't have captured anything useful yet.
    • Upload: from the logs screen, tap Upload Logs, and add a note identifying the device so it can be matched up when you report it.
    • Disable afterward: clear the same Extra tags for logging field and confirm — otherwise it keeps generating a lot of extra log volume.
